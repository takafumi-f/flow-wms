## 1. ルートレベルの設定ファイル群

### README.md / .gitignore / .env.example

| ファイル | 役割 |
|----------|------|
| `README.md` | プロジェクト概要、ローカル起動手順、環境変数一覧、アーキテクチャ概説を記載する |
| `.gitignore` | `node_modules/`, `.next/`, `.env.local`, `*.log`, `dist/` 等をバージョン管理対象外とする |
| `.env.example` | 実際の秘密情報を含まないキーの一覧テンプレート。`DATABASE_URL`, `REDIS_URL`, `NEXTAUTH_SECRET`, `NEXTAUTH_URL` 等を定義する |

`.env.example` には必ず全環境変数キーを列挙し、値にはプレースホルダー（例: `DATABASE_URL=postgresql://user:password@localhost:5432/wms`）を記載する。`.env.local` は `.gitignore` で除外し、リポジトリに含めない。

### docker-compose.yml

ローカル開発環境の PostgreSQL と Redis を一括起動するための構成ファイル。

```yaml
# 主要サービス定義（概要）
services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_DB: wms
      POSTGRES_USER: wms_user
      POSTGRES_PASSWORD: wms_password
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - ./db/schema.sql:/docker-entrypoint-initdb.d/01_schema.sql
      - ./db/seed:/docker-entrypoint-initdb.d/seed

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
```

`docker-entrypoint-initdb.d/` に `schema.sql` と seed ファイルをマウントすることで、コンテナ初回起動時に自動的にスキーマとデモデータが投入される。

### next.config.ts

```typescript
// 重要な設定項目
const config: NextConfig = {
  // PWA対応（ハンディ端末向け）
  // next-pwa または @ducanh2912/next-pwa を使用
  experimental: {
    // Server Actions を明示的に有効化（Next.js 15 では stable だが確認推奨）
    typedRoutes: true,  // リンクの型安全性
  },
  // 画像ドメイン許可設定
  images: {
    remotePatterns: [],
  },
};
```

**Next.js 15 の注意点:**
- `fetch()` のキャッシュ動作がデフォルト `no-store` に変更された。意図的にキャッシュしたい場合は `{ cache: 'force-cache' }` または `next: { revalidate: N }` を明示する。
- `cookies()` / `headers()` 等の Dynamic API は非同期（`await` 必須）に変更された。
- `params` と `searchParams` が Promise になったため、Page コンポーネントでは `await props.params` のように扱う。

### tsconfig.json

```jsonc
{
  "compilerOptions": {
    "target": "ES2022",
    "lib": ["dom", "dom.iterable", "esnext"],
    "strict": true,              // 必須：型安全性を最大化
    "noUncheckedIndexedAccess": true,  // 配列アクセスの undefined 考慮

      "@/*": ["./src/*"]         // パスエイリアス
```

## 2. db/ ディレクトリの管理方針

### db/schema.sql

全テーブルの DDL（CREATE TABLE, CREATE INDEX, CREATE TYPE 等）を単一ファイルで管理する。このファイルは常に「現在の最新スキーマ」を表す正規定義ファイルであり、docker-compose 初回起動時に自動適用される。

管理方針：
- テーブル定義の変更は `schema.sql` への直接編集 **と** `migrations/` へのマイグレーションファイル追加を **両方** 行う。
- `schema.sql` は `DROP TABLE IF EXISTS` を使わず、初期構築専用として管理する（冪等性より可読性を優先）。
- 本番環境では `schema.sql` を使わず `migrations/` を順番に適用する。

### db/migrations/

連番プレフィックスでマイグレーション SQL を管理する。

```
migrations/
├── 001_initial_schema.sql
├── 002_add_lot_expiry_to_inventory.sql
├── 003_add_index_shipping_status.sql
└── ...
```

- ファイル名フォーマット: `NNN_description.sql`（N は3桁ゼロ埋め連番）。
- 各ファイルはべき等に設計する（`IF NOT EXISTS`, `IF EXISTS` 句を活用）。
- マイグレーション管理ツール（`dbmate` や `golang-migrate` 等）の導入を推奨するが、`package.json` の `db:migrate` スクリプトで連番 SQL を順番に実行する簡易方式でも可。
- **ロールバック SQL は別ファイル（`NNN_description.down.sql`）として用意する。**

### db/seed/

| ファイル | 内容 |
|----------|------|
| `01_master.sql` | 商品マスタ、取引先マスタ、ユーザーマスタ、ロケーションマスタの基本データ。本番環境の初期データとしても使用可能な範囲に留める。 |
| `02_demo.sql` | デモ・開発環境専用のサンプルデータ（入荷伝票、出荷伝票、在庫数量等）。本番環境では適用しない。 |

seed ファイルは `01_master.sql` → `02_demo.sql` の順で適用する。`02_demo.sql` は `01_master.sql` のマスタデータに依存する外部キーを持つため、順序の入れ替えは禁止。

## 3. src/app/ の Route Groups の意図

### (auth) グループ

`src/app/(auth)/login/page.tsx`

認証関連ページを格納する Route Group。括弧付きディレクトリ名はURLパスに影響しないため、`/login` というURLを維持しつつレイアウトを分離できる。

- `(auth)` グループには独自の `layout.tsx` を設けない（または最小限のもの）。ログイン画面はサイドバーやナビゲーションを表示しないシンプルな全画面レイアウトとなる。
- `(app)` グループの `layout.tsx` に認証チェック（`middleware.ts` または Server Component での `getServerSession` 確認）を実装し、未認証ユーザーを `/login` にリダイレクトする。

### (app) グループ

`src/app/(app)/layout.tsx` - サイドバー・ヘッダーを含む共通レイアウト

認証済みユーザー向けの全業務画面を格納する Route Group。`layout.tsx` でサイドバー・ヘッダー等の共通ナビゲーションをレンダリングする。

```
(app)/
├── layout.tsx        # 認証チェック + サイドバー + ヘッダー（Server Component）
├── page.tsx          # KPIダッシュボード（/）
├── receiving/        # /receiving/* - 入荷管理
├── shipping/         # /shipping/* - 出荷管理
├── inventory/        # /inventory/* - 在庫管理
├── location/         # /location/* - ロケーション管理
├── master/           # /master/* - マスタ管理
└── reports/          # /reports/* - レポート
```

各業務ドメインのルーティング設計：

| パス | 役割 |
|------|------|
| `/receiving` | 入荷伝票一覧 |
| `/receiving/new` | 新規入荷伝票作成 |
| `/receiving/[id]` | 入荷伝票詳細 |
| `/receiving/[id]/edit` | 入荷伝票編集 |
| `/receiving/[id]/inspect` | 入荷検品作業画面 |
| `/shipping/[id]/picking` | ピッキング作業画面 |
| `/inventory/move` | 在庫移動 |
| `/inventory/stocktake` | 棚卸 |
| `/inventory/trends` | 在庫推移グラフ |

### handy/ グループ（PWA）

`src/app/handy/` - ハンディ端末専用 Route（Route Group ではなく通常ディレクトリ）

倉庫作業員がハンディ端末（スマートフォン・専用ハンディターミナル）で使用するシンプルな画面群。

設計方針：
- `handy/layout.tsx` はサイドバーを持たない全画面レイアウト。タップターゲットは最低 56px を確保。
- フォントサイズは最低 18px 以上、ボタンは画面幅いっぱいを活用する。
- PWA マニフェストを設定し、ホーム画面追加・オフラインキャッシュ対応を実現する。
- ハンディ端末向けページは Client Component を主体とし、バーコードスキャン（`scanner/` コンポーネント）との連携を前提とする。
- 認証は `(app)` と同じセッション管理を使用するが、ハンディ端末用の認証フローを別途検討する（QRコードログイン等）。

```
/handy/picking    # ピッキング作業（商品スキャン → 数量確認）
/handy/receiving  # 入荷検品（バーコードスキャン → 棚入れ指示）
```

## 4. src/app/api/ の Route Handler 設計方針

### 基本方針

Next.js 15 の Route Handler（`route.ts`）を REST API として設計する。各 `route.ts` ファイルは対応する HTTP メソッドをエクスポートする。

```typescript
// 典型的な route.ts の構造
export async function GET(request: Request) { ... }
export async function POST(request: Request) { ... }
```

### 認証

```
api/auth/[...nextauth]/route.ts
```

NextAuth v5 の catch-all ルートハンドラ。GET・POST をエクスポートし、ログイン・ログアウト・セッション確認を処理する。全ての業務 API エンドポイントではこのセッションを検証してから処理を行う。

### エンドポイント設計

| エンドポイント | メソッド | 役割 |
|--------------|---------|------|
| `/api/receiving/orders` | GET | 入荷伝票一覧取得（フィルタ・ページネーション対応） |
| `/api/receiving/orders` | POST | 入荷伝票新規作成 |
| `/api/receiving/orders/[id]` | GET | 入荷伝票詳細取得 |
| `/api/receiving/orders/[id]` | PATCH | 入荷伝票更新（ステータス変更含む） |
| `/api/receiving/orders/[id]` | DELETE | 入荷伝票削除（論理削除） |
| `/api/shipping/orders` | GET / POST | 出荷伝票一覧・新規作成 |
| `/api/shipping/orders/[id]` | GET / PATCH | 出荷伝票詳細・更新 |
| `/api/inventory` | GET | 在庫一覧（ロケーション・商品でフィルタ） |
| `/api/inventory/move` | POST | 在庫移動処理（トランザクション） |
| `/api/inventory/alerts` | GET | 欠品・過剰在庫アラート一覧 |
| `/api/locations` | GET / POST / PATCH | ロケーション管理 |
| `/api/master/items` | GET / POST / PATCH | 商品マスタ管理 |
| `/api/master/partners` | GET / POST / PATCH | 取引先マスタ管理 |
| `/api/master/users` | GET / POST / PATCH | ユーザーマスタ管理 |
| `/api/reports/kpi` | GET | KPIダッシュボード集計データ |
| `/api/reports/operations` | GET | 作業実績レポートデータ |

### レスポンス形式

全エンドポイントで統一したレスポンス形式を採用する：

```typescript
// 成功
{ data: T, meta?: { total: number, page: number } }

// エラー
{ error: { code: string, message: string, details?: unknown } }
```

### Next.js 15 の注意点

- `request.json()` は `Promise<unknown>` を返す。Zod でバリデーションしてから使用する。
- Route Handler 内での `cookies()` / `headers()` 呼び出しは `await` が必要（Next.js 15 から非同期化）。
- `NextResponse.json()` はデフォルトでキャッシュされない。在庫データ等はキャッシュが不要なため問題ないが、KPIレポートは `revalidate` を検討する。
- 動的パラメータの型は `Promise<{ id: string }>` となるため、`await params` で解決してから使用する：

```typescript
export async function GET(
  request: Request,
```

## 5. src/components/ の分類基準とコンポーネント設計方針

### Server Component vs Client Component の判断基準

| 基準 | Server Component | Client Component |
|------|-----------------|-----------------|
| データフェッチ | DB・API から直接取得 | SWR / fetch で取得 |
| インタラクティビティ | 不要（表示のみ） | 必要（フォーム、クリック等） |
| ブラウザ API | 使用しない | 使用する（カメラ、LocalStorage 等） |
| 状態管理 | 不要 | useState / useReducer 使用 |

**原則: できる限り Server Component とし、インタラクティビティが必要な最小単位のみ Client Component（`"use client"` 指定）にする。**

### ui/ - 共通UIプリミティブ

```
ui/
├── Button.tsx       # バリアント（primary/secondary/danger）+ サイズ + ローディング状態
├── Card.tsx         # コンテナカード（ヘッダー・ボディ・フッター）
├── Table.tsx        # ソート・ページネーション対応テーブル
├── Badge.tsx        # ステータスバッジ（色はステータスに連動）
├── Input.tsx        # テキスト入力（バリデーションエラー表示対応）
├── Select.tsx       # セレクトボックス
├── Modal.tsx        # モーダルダイアログ（"use client"）
├── Toast.tsx        # トースト通知（"use client"）
├── Spinner.tsx      # ローディングスピナー
└── EmptyState.tsx   # データなし状態の表示
```

- shadcn/ui や Radix UI をベースとし、WMS 固有のデザイントークン（ステータスカラー等）を適用する。
- `ui/` のコンポーネントはビジネスロジックを持たない純粋な表示コンポーネントとする。
- `Badge` のステータス色は `tailwind.config.ts` で定義したデザイントークンを使用し、ハードコーディングしない。

### schemas.ts - Zodバリデーションスキーマ

```typescript
  expectedDate: z.coerce.date(),
  lines: z.array(z.object({
    itemId: z.string().uuid(),
```

## 7. src/types/ の型定義方針

### types/index.ts

全ドメインの型定義を単一ファイルで管理する。プロジェクト規模が小さい段階は集約管理し、規模拡大時にファイル分割を検討する。

```typescript
// ===== 共通 =====
export type UUID = string;
export type ISODateString = string;

export type Status<T extends string> = T;

// ===== ユーザー =====
export type UserRole = "admin" | "operator" | "viewer";

export type User = {
  id:        UUID;
  name:      string;
  email:     string;
  role:      UserRole;
  createdAt: ISODateString;
};

// ===== 商品マスタ =====
export type Item = {
  id:          UUID;
  code:        string;      // 商品コード（バーコード）
  name:        string;
  unit:        string;      // 単位（個、箱、kg 等）
  minStock:    number;      // 最低在庫数（アラート閾値）
  maxStock:    number | null;
  isActive:    boolean;
};

// ===== ロケーション =====
export type Location = {
  id:           UUID;
  code:         string;     // 例: "A-01-001"
  area:         string;
  capacity:     number | null;
  isActive:     boolean;
};

// ===== 在庫 =====
export type InventoryStockStatus = "normal" | "low" | "out" | "excess";

export type InventoryStock = {
  itemId:      UUID;
  item:        Item;
  locationId:  UUID;
  location:    Location;
  quantity:    number;
  status:      InventoryStockStatus;
  updatedAt:   ISODateString;
};

// ===== 入荷 =====
export type ReceivingOrderStatus = "draft" | "confirmed" | "inspecting" | "completed" | "cancelled";

export type ReceivingOrder = {
  id:            UUID;
  orderNo:       string;
  partnerId:     UUID;
  status:        ReceivingOrderStatus;
  expectedDate:  ISODateString;
  completedAt:   ISODateString | null;
  lines:         ReceivingOrderLine[];
  memo:          string | null;
  createdBy:     UUID;
  createdAt:     ISODateString;
};

export type ReceivingOrderLine = {
  id:              UUID;
  orderId:         UUID;
  itemId:          UUID;
  item:            Item;
  orderedQty:      number;
  receivedQty:     number;
  locationId:      UUID | null;
  location:        Location | null;
  inspectedAt:     ISODateString | null;
};

// ===== 出荷 =====
export type ShippingOrderStatus = "draft" | "confirmed" | "picking" | "inspecting" | "shipped" | "cancelled";

export type ShippingOrder = {
  id:          UUID;
  orderNo:     string;
  partnerId:   UUID;
  status:      ShippingOrderStatus;
  shipDate:    ISODateString;
  shippedAt:   ISODateString | null;
  lines:       ShippingOrderLine[];
  memo:        string | null;
  createdBy:   UUID;
  createdAt:   ISODateString;
};

export type ShippingOrderLine = {
  id:            UUID;
  orderId:       UUID;
  itemId:        UUID;
  item:          Item;
  requiredQty:   number;
  pickedQty:     number;
  locationId:    UUID | null;
  pickedAt:      ISODateString | null;
};
```

型定義の方針：
- DB の `null` 可能なカラムは TypeScript の `null`（`undefined` ではなく）で表現する。
- エンティティの JOIN 結果（`item` を含む `ReceivingOrderLine` 等）はネストした型で定義し、APIレスポンスの型として使用する。
- ステータス型は `union type` で定義し、文字列リテラルによる型安全を確保する。
- `UUID` / `ISODateString` 等のブランド型エイリアスで意図を明示する（strict なブランド型が必要な場合は `& { readonly _brand: "UUID" }` を検討）。

```typescript
describe("allocateByFifo", () => {
  it("古いロットから優先的に引き当てる", () => {
    const lots: InventoryLot[] = [
      { lotId: "lot1", receivedAt: new Date("2026-01-01"), quantity: 10, locationId: "loc1" },
      { lotId: "lot2", receivedAt: new Date("2026-01-15"), quantity: 20, locationId: "loc1" },

  it("在庫不足の場合は shortage に不足数を返す", () => {
    const lots: InventoryLot[] = [
      { lotId: "lot1", receivedAt: new Date("2026-01-01"), quantity: 5, locationId: "loc1" },
    ];
    const result = allocateByFifo(lots, 10);
    expect(result.shortage).toBe(5);
  });
});
```

## 実装全体の注意事項

### Next.js 15 対応チェックリスト

- [ ] `cookies()` / `headers()` / `params` / `searchParams` の `await` 忘れに注意
- [ ] `fetch()` のキャッシュがデフォルト `no-store` になったことを前提にキャッシュ戦略を見直す
- [ ] Server Actions のバリデーションは `schemas.ts` の Zod スキーマで必ず実施する
- [ ] `"use client"` を付与したコンポーネントは独立してバンドルされる（バンドルサイズを意識する）

### TypeScript 型安全性チェックリスト

- [ ] `strict: true` と `noUncheckedIndexedAccess: true` が有効になっていること
- [ ] APIレスポンスの型は `types/index.ts` で定義した型を使い、`any` を使用しない
- [ ] DB クエリ結果は `as T` でキャストせず、Zod でバリデーションしてから型を付与する
- [ ] 環境変数は `!` 演算子でのアサーションを最小限にし、起動時にバリデーションする

### パフォーマンス考慮事項

- [ ] 在庫一覧ページは無限スクロールまたはサーバーサイドページネーションを実装し、全件取得を避ける
- [ ] ダッシュボードの KPI は Redis キャッシュを活用し、毎リクエスト DB 集計しない
- [ ] `location/` のフロアマップは SVG を静的に生成し、在庫数のみを動的更新する
- [ ] `recharts` 等のグラフライブラリは動的インポート（`next/dynamic`）でコード分割する
