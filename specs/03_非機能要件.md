# FLOW WMS 非機能要件仕様書

**文書番号**: FLOW-WMS-SPEC-003  
**バージョン**: 1.0  
**作成日**: 2026-06-24  
**ステータス**: ドラフト

## 1. パフォーマンス要件

### 1.1 画面応答時間

| 要件ID | 操作種別 | 目標応答時間 | 測定条件 |
|--------|---------|------------|---------|
| NF-PERF-01 | 通常画面操作（画面遷移・ボタン操作・フォーム送信） | **1秒以内** | 同時接続50ユーザー時、95パーセンタイル |
| NF-PERF-02 | 検索・一覧表示（在庫照会・入出荷履歴等） | **3秒以内** | 同時接続100ユーザー時、95パーセンタイル |
| NF-PERF-03 | KPIダッシュボード初期表示 | **2秒以内** | 同時接続50ユーザー時、95パーセンタイル |
| NF-PERF-04 | PDF帳票生成・ダウンロード | **5秒以内** | A4 5ページ相当 |

### 1.2 スキャン処理応答時間

| 要件ID | 要件内容 |
|--------|---------|
| NF-PERF-05 | バーコード/QRスキャン後の在庫照会・検品照合処理は **500ms以内** に完了すること |
| NF-PERF-06 | ハンディターミナルのスキャン入力から次の作業指示表示まで **500ms以内** であること |
| NF-PERF-07 | スキャン処理の応答時間は端末側の解析処理を含まないサーバーサイド処理時間として計測すること |

### 1.3 スループット・同時接続

| 要件ID | 要件内容 |
|--------|---------|
| NF-PERF-08 | 同時接続ユーザー数 **100ユーザー** において、全パフォーマンス要件を満たすこと |
| NF-PERF-09 | 在庫照会APIのスループットは **200 req/s** を維持できること |
| NF-PERF-10 | ピーク時（同時接続100ユーザー）においてもCPU使用率 80%以下を維持すること |
| NF-PERF-11 | WebSocket接続は **100コネクション** を同時に維持できること |

### 1.4 データ量に関する前提条件

| 項目 | 前提値 |
|------|--------|
| 品目マスタ件数 | 最大 100,000件 |
| ロケーション件数 | 最大 50,000件 |
| 在庫レコード件数 | 最大 500,000件 |
| 日次入出荷件数 | 最大 10,000件/日 |
| 保管ログ期間 | 5年間 |

## 2. 可用性要件

### 2.1 稼働率目標

| 要件ID | 要件内容 |
|--------|---------|
| NF-AVAIL-01 | システム稼働率は **99.9%以上** を維持すること（年間停止許容時間: **8.7時間以内**） |
| NF-AVAIL-02 | 稼働率は計画メンテナンス時間を除いた実稼働時間で算出すること |
| NF-AVAIL-03 | 障害発生からサービス復旧までの目標復旧時間（RTO）は **2時間以内** とすること |
| NF-AVAIL-04 | 障害発生時のデータ損失許容時間（RPO）は **1時間以内** とすること |

### 2.2 計画メンテナンス

| 要件ID | 要件内容 |
|--------|---------|
| NF-AVAIL-05 | 計画メンテナンス時間は **毎週日曜日 02:00〜05:00**（最大3時間）とすること |
| NF-AVAIL-06 | 計画メンテナンスの実施は1週間前にシステム内通知・メール通知を行うこと |
| NF-AVAIL-07 | 緊急メンテナンスが必要な場合は最低12時間前に利用者へ通知すること |

### 2.3 フェイルオーバー・冗長化

| 要件ID | 要件内容 |
|--------|---------|
| NF-AVAIL-08 | PostgreSQL はプライマリ-レプリカ構成（ストリーミングレプリケーション）とし、プライマリ障害時は自動フェイルオーバーを行うこと |
| NF-AVAIL-09 | アプリケーションサーバーは最低2台構成とし、1台障害時もサービスを継続できること |
| NF-AVAIL-10 | Redis はレプリカ構成（Redis Sentinel または Redis Cluster）とし、障害時のサービス継続を保証すること |
| NF-AVAIL-11 | ロードバランサーはヘルスチェックを実施し、異常サーバーを自動で切り離すこと（ヘルスチェック間隔: 10秒） |
| NF-AVAIL-12 | フェイルオーバー完了後、進行中のトランザクションが整合性を維持すること（データ不整合ゼロ） |

## 3. セキュリティ要件

### 3.1 認証

| 要件ID | 要件内容 |
|--------|---------|
| NF-SEC-01 | ユーザー認証にJWT（JSON Web Token）とサーバーサイドセッション（NextAuth v5）を組み合わせて実装すること |
| NF-SEC-02 | JWTのアクセストークン有効期限は **15分**、リフレッシュトークン有効期限は **7日** とすること |
| NF-SEC-03 | 2要素認証（2FA）に対応し、TOTP（Google Authenticator等）によるワンタイムパスワードを利用できること |
| NF-SEC-04 | ログイン試行失敗 **5回** でアカウントを **30分** ロックする機能を実装すること |
| NF-SEC-05 | セッションタイムアウト（無操作 **60分**）後は再認証を要求すること |
| NF-SEC-06 | ハンディターミナルからのアクセスにも同一認証機構を適用すること |

### 3.2 ロールベースアクセス制御（RBAC）

| 要件ID | 要件内容 |
|--------|---------|
| NF-SEC-07 | 全APIエンドポイント・画面にロールベースのアクセス制御（RBAC）を実装すること |
| NF-SEC-08 | ロールは「システム管理者・倉庫管理者・入出荷担当者・ピッカー」の4種類を基本とし、カスタムロールを追加できること |
| NF-SEC-09 | 権限のないリソースへのアクセス試行は HTTP 403 を返し、アクセスログに記録すること |
| NF-SEC-10 | マスタ管理・ユーザー管理へのアクセスはシステム管理者ロールのみに制限すること |

### 3.3 通信暗号化

| 要件ID | 要件内容 |
|--------|---------|
| NF-SEC-11 | すべてのHTTP通信はTLS **1.3** で暗号化すること（TLS 1.2以下は無効化） |
| NF-SEC-12 | HSTSヘッダーを設定し、HTTP→HTTPSリダイレクトを強制すること |
| NF-SEC-13 | WebSocket通信はWSSプロトコル（TLS上のWebSocket）を使用すること |
| NF-SEC-14 | 本番証明書はLet's Encryptまたは認証局発行の証明書を使用し、有効期限を自動更新すること |
| NF-SEC-15 | 内部サービス間通信（アプリ-DB・アプリ-Redis）もTLSまたはVPN内通信で保護すること |

### 3.4 監査ログ

| 要件ID | 要件内容 |
|--------|---------|
| NF-SEC-16 | ユーザーの全操作（ログイン・ログアウト・データ作成・更新・削除・マスタ変更）を監査ログとして記録すること |
| NF-SEC-17 | 監査ログには操作日時・ユーザーID・IPアドレス・操作種別・対象データ（変更前後）を含めること |
| NF-SEC-18 | 監査ログは改ざん防止のため、アプリケーション側からは削除・更新不可とすること |
| NF-SEC-19 | 監査ログの保存期間は **5年間** とすること |
| NF-SEC-20 | 監査ログはシステム管理者がUI上で検索・参照できること |

### 3.5 データ暗号化

| 要件ID | 要件内容 |
|--------|---------|
| NF-SEC-21 | データベース（PostgreSQL）は透過的データ暗号化（TDE）または暗号化ファイルシステムを適用すること |
| NF-SEC-22 | パスワードはbcryptまたはArgon2でハッシュ化して保存すること（平文保存禁止） |
| NF-SEC-23 | APIキー・外部システム接続情報等の機密情報は環境変数または Secret Manager で管理し、ソースコードに埋め込まないこと |
| NF-SEC-24 | バックアップファイルは暗号化した状態でオブジェクトストレージに保存すること |

### 3.6 その他セキュリティ

| 要件ID | 要件内容 |
|--------|---------|
| NF-SEC-25 | OWASP Top 10 の脅威に対する対策を実装すること（SQLインジェクション・XSS・CSRF・セキュリティヘッダー設定等） |
| NF-SEC-26 | 定期的な依存パッケージの脆弱性スキャン（npm audit / Dependabot）を実施すること |
| NF-SEC-27 | 入力値のバリデーション（Zod）をサーバーサイドで実施すること |

## 4. スケーラビリティ要件

### 4.1 水平スケーリング

| 要件ID | 要件内容 |
|--------|---------|
| NF-SCALE-01 | アプリケーション層（Next.js）はステートレス設計とし、サーバー台数の追加（水平スケーリング）に対応すること |
| NF-SCALE-02 | セッション情報はRedisに集中管理し、複数アプリサーバー間でセッション共有できること |
| NF-SCALE-03 | コンテナ（Docker）ベースのデプロイとし、オーケストレーション（Kubernetes / Docker Compose）に対応できること |
| NF-SCALE-04 | 将来的な利用拡大に対応するため、設計時点で同時接続1,000ユーザーへのスケールアップを想定した設計とすること |

### 4.2 DBコネクションプール

| 要件ID | 要件内容 |
|--------|---------|
| NF-SCALE-05 | PostgreSQLへの接続管理にPgBouncerを使用し、コネクションプールを実現すること |
| NF-SCALE-06 | PgBouncerのプールモードは **Transaction Pooling** を使用すること |
| NF-SCALE-07 | PgBouncerの最大コネクション数をPostgreSQLの `max_connections` の80%以内に設定すること |
| NF-SCALE-08 | DBコネクション数・待機キュー長を監視し、閾値超過時にアラートを発報すること |

### 4.3 キャッシュ戦略

| 要件ID | 要件内容 |
|--------|---------|
| NF-SCALE-09 | Redisを利用したキャッシュ戦略を実装すること |
| NF-SCALE-10 | 品目マスタ・ロケーションマスタ・ユーザー権限情報を **TTL 300秒** でRedisにキャッシュすること |
| NF-SCALE-11 | KPIダッシュボードの集計データを **TTL 60秒** でRedisにキャッシュすること |
| NF-SCALE-12 | 在庫数量はリアルタイム性が求められるため、キャッシュを使用せず常にDBから取得すること |
| NF-SCALE-13 | キャッシュ更新時はキャッシュパージ（Invalidation）を適切に実施し、データ不整合を防止すること |

### 4.4 CDN・静的アセット配信

| 要件ID | 要件内容 |
|--------|---------|
| NF-SCALE-14 | JavaScript・CSS・画像等の静的アセットをCDN（Cloudflare等）経由で配信すること |
| NF-SCALE-15 | Next.jsの静的ファイルにはCache-Controlヘッダー（immutable, max-age=31536000）を設定すること |
| NF-SCALE-16 | CDNキャッシュの purge（パージ）はデプロイパイプラインから自動実行すること |

## 5. 保守性要件

### 5.1 コード品質

| 要件ID | 要件内容 |
|--------|---------|
| NF-MAINT-01 | ユニットテスト・結合テストのコードカバレッジは **80%以上** を維持すること |
| NF-MAINT-02 | ユニットテストフレームワークにVitestを使用すること |
| NF-MAINT-03 | E2EテストフレームワークにPlaywrightを使用し、主要ユーザーフローをカバーすること |
| NF-MAINT-04 | ESLintおよびPrettierを導入し、コーディング規約を自動チェック・自動整形すること |
| NF-MAINT-05 | TypeScriptの `strict` モードを有効にし、型安全性を確保すること |

### 5.2 CI/CD パイプライン

| 要件ID | 要件内容 |
|--------|---------|
| NF-MAINT-06 | GitHub Actionsを使用したCI/CDパイプラインを構築すること |
| NF-MAINT-07 | プルリクエスト作成時に自動でリント・型チェック・ユニットテスト・ビルドを実行すること |
| NF-MAINT-08 | mainブランチへのマージ後、自動でステージング環境へデプロイすること |
| NF-MAINT-09 | 本番環境へのデプロイは承認フロー（手動承認）を経て実行すること |
| NF-MAINT-10 | デプロイ失敗時は自動でロールバックし、Slackへ通知すること |
| NF-MAINT-11 | ゼロダウンタイムデプロイ（ローリングアップデート）を実現すること |

### 5.3 ログ管理

| 要件ID | 要件内容 |
|--------|---------|
| NF-MAINT-12 | アプリケーションログはJSON形式の構造化ログとして出力すること |
| NF-MAINT-13 | ログレベルは環境変数で制御できること（DEBUG / INFO / WARN / ERROR） |
| NF-MAINT-14 | ログには必ずリクエストID（トレースID）を含め、リクエストの追跡を可能にすること |
| NF-MAINT-15 | エラーログにはスタックトレース・ユーザーID・エンドポイント・レスポンスタイムを含めること |
| NF-MAINT-16 | ログの保存期間は **INFO以上: 90日、ERROR: 1年** とすること |
| NF-MAINT-17 | 本番環境のDEBUGログはデフォルト無効とし、デバッグ時のみ有効化できること |

### 5.4 モニタリング

| 要件ID | 要件内容 |
|--------|---------|
| NF-MAINT-18 | Prometheusを使用したメトリクス収集を実装すること |
| NF-MAINT-19 | 収集するメトリクスにはHTTPリクエスト数・レスポンスタイム・エラー率・DBクエリ時間・Redisキャッシュヒット率を含めること |
| NF-MAINT-20 | GrafanaでPrometheusのメトリクスをダッシュボード表示すること |
| NF-MAINT-21 | 以下の条件でアラートを発報すること：<br>- エラー率 1%超（5分間） <br>- レスポンスタイム P95 が 3秒超（5分間）<br>- DB接続エラー（即時）<br>- サーバーCPU使用率 90%超（10分間）<br>- ディスク使用率 80%超 |
| NF-MAINT-22 | アラートはSlackチャンネルおよびメールへ通知すること |

### 5.5 バックアップ

| 要件ID | 要件内容 |
|--------|---------|
| NF-MAINT-23 | PostgreSQLのフルバックアップを **毎日 01:00** に実行し、暗号化の上オブジェクトストレージへ保存すること |
| NF-MAINT-24 | PostgreSQL WAL（Write-Ahead Logging）を使用したPITR（Point-in-Time Recovery）に対応すること |
| NF-MAINT-25 | バックアップデータの保存期間は **30日間** とすること |
| NF-MAINT-26 | バックアップからのリストアを月次で手順確認し、RTO目標（2時間）を満たすことを検証すること |
| NF-MAINT-27 | バックアップ成功・失敗をモニタリングし、失敗時はアラートを発報すること |

## 6. 対応環境

### 6.1 PCブラウザ

| 要件ID | 要件内容 |
|--------|---------|
| NF-ENV-01 | 対応ブラウザはGoogle Chrome・Microsoft Edgeの各 **最新2バージョン** とすること |
| NF-ENV-02 | Internet Explorer・旧版Safari（iOS未満のバージョン）は対象外とすること |
| NF-ENV-03 | JavaScriptを無効にした環境での動作は保証しないこと |
| NF-ENV-04 | ブラウザ拡張機能による表示崩れへの対応は対象外とすること |

### 6.2 PC解像度・レスポンシブ対応

| 要件ID | 要件内容 |
|--------|---------|
| NF-ENV-05 | PC画面は解像度 **1280×768以上** で正常に表示されること |
| NF-ENV-06 | 推奨解像度はフルHD（1920×1080）とすること |
| NF-ENV-07 | 管理画面（PC向け）はレスポンシブデザインを適用し、1280px未満の画面幅には対応しないこと |

### 6.3 ハンディターミナル（PWA）

| 要件ID | 要件内容 |
|--------|---------|
| NF-ENV-08 | ハンディターミナル向けUIはPWA（Progressive Web App）として提供すること |
| NF-ENV-09 | 対応OSはAndroid **10以上** とすること |
| NF-ENV-10 | ハンディ向け画面はタッチ操作・グローブ着用での操作を考慮し、タップ領域を最小 **44×44px** 以上とすること |
| NF-ENV-11 | ハンディ向け画面はポートレート（縦向き）表示を基本とすること |
| NF-ENV-12 | 一時的なWiFi切断（30秒以内）時は作業継続し、復旧後に自動同期すること（オフライン対応） |
| NF-ENV-13 | PWAのサービスワーカーにより、最後に受信した作業指示データをキャッシュすること |

### 6.4 フォークリフト端末

| 要件ID | 要件内容 |
|--------|---------|
| NF-ENV-14 | フォークリフト端末向けUIはPWAとして提供し、10インチ以上のタブレットで使用できること |
| NF-ENV-15 | 倉庫環境（強い日差し・振動）での視認性を考慮し、高コントラスト・大きなフォントサイズ（最小16px）を使用すること |
| NF-ENV-16 | ランドスケープ（横向き）表示を基本とすること |

## 7. 国際化・ローカライゼーション要件

| 要件ID | 要件内容 |
|--------|---------|
| NF-I18N-01 | システムの表示言語は **日本語** を基本言語とすること |
| NF-I18N-02 | 将来的な多言語対応（英語・中国語）を見据え、テキストをi18nリソースファイルで管理すること |
| NF-I18N-03 | 日時フォーマットは **日本標準時（JST/UTC+9）** を使用すること |
| NF-I18N-04 | 数値フォーマットは日本の慣習（桁区切りカンマ）に従うこと |

## 8. データ保護・コンプライアンス要件

| 要件ID | 要件内容 |
|--------|---------|
| NF-COMP-01 | 個人情報（ユーザー情報・取引先担当者情報）の取り扱いは個人情報保護法に準拠すること |
| NF-COMP-02 | 個人情報を含む画面・APIへのアクセスは認証・認可を必須とすること |
| NF-COMP-03 | 在庫・取引データは法定保存期間（商業帳簿: 10年）を考慮したデータ保管方針を設けること |
| NF-COMP-04 | システム廃止時のデータ移行・削除に関する手順を運用マニュアルに定めること |
