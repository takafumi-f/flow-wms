## 2. 仕様書のバージョン管理方針

### バージョニング規則

セマンティックバージョニング（`Major.Minor.Patch`）を採用します。

| 区分 | 説明 | 例 |
|---|---|---|
| **Major**（メジャー） | 既存仕様との後方互換性がない破壊的変更 | v1.0.0 → v2.0.0 |
| **Minor**（マイナー） | 後方互換性を維持した機能追加・仕様拡張 | v1.0.0 → v1.1.0 |
| **Patch**（パッチ） | 誤記修正・表現の明確化など軽微な変更 | v1.0.0 → v1.0.1 |

### 変更管理プロセス

```
変更申請 → レビュー → 承認 → 更新・マージ
```

1. **変更申請**: 変更内容・理由・影響範囲を記載した Issue を GitHub に起票する
2. **レビュー**: 担当者（設計者・ステークホルダー）による内容確認と議論
3. **承認**: プロジェクトリードによる最終承認
4. **更新**: 仕様書ファイルを修正し、更新履歴テーブルを記入してプルリクエストを作成
5. **マージ**: 承認後に main ブランチへマージ

### Git 管理方針

**ブランチ戦略**

| ブランチ | 用途 |
|---|---|
| `main` | 承認済みの最新仕様書を管理する安定ブランチ |
| `docs/feature/<機能名>` | 新規仕様書・大規模改訂の作業ブランチ |
| `docs/fix/<修正内容>` | 誤記・軽微な修正の作業ブランチ |

**コミットメッセージ規約**

```
<種別>(<対象ファイル>): <変更内容の概要>

例:
feat(06_API設計): 在庫照会エンドポイントを追加
fix(03_データベース設計): productsテーブルのカラム定義の誤記を修正
docs(README): バージョン管理方針を追記
```

### 変更影響範囲の分類

| 分類 | 内容 | 対応バージョン |
|---|---|---|
| 破壊的変更 | API仕様変更、テーブル定義の削除・大幅変更など、既存実装への影響が大きい変更 | Major |
| 機能追加 | 新規エンドポイント追加、新テーブル追加など、後方互換性を維持した仕様拡張 | Minor |
| 軽微な修正 | 誤字脱字の修正、説明文の補足・明確化、フォーマット整理 | Patch |

## 4. 読み方ガイド

新規参画者は以下の順序で仕様書を読むことを推奨します。

### ステップ 1: 全体像の把握（1〜2時間）

1. **01_システム概要.md** — プロダクトコンセプト・アーキテクチャ・市場戦略・SaaSビジネスモデルを把握する。特に「セクション10（市場戦略）」はなぜこのシステムを作るかを理解するために必読。

### ステップ 2: 機能・設計の理解（2〜3時間）

2. **02_機能要件.md** — F-01〜F-48 の全機能要件を把握する。EC連携（F-41〜F-43）・3PL対応（F-44〜F-46）・SaaS対応（F-47〜F-48）が重要。
3. **05_DB設計.md** — データモデルと各テーブルの関係を把握する。システムの骨格となる情報。
4. **09_EC連携設計.md** — Amazon/楽天/Shopifyとの連携、ヤマト/佐川/ゆうパックの送り状発行設計を把握する。FLOW WMSの核心的差別化機能。

### ステップ 3: 実装詳細の確認（2〜3時間）

5. **06_API設計.md** — REST API の全エンドポイントとリクエスト/レスポンス仕様を確認する。
6. **07_システム連携設計.md** — ERP・EDI・ハンディ端末・AGV・ラベルプリンタとの連携仕様を確認する。
7. **04_画面設計.md** — 画面設計とユーザー操作フローを確認する。
8. **08_ディレクトリ構成.md** — 実装コードのファイル構成と各ディレクトリの役割を確認し、開発環境に慣れる。
9. **03_非機能要件.md** — パフォーマンス・可用性・セキュリティ要件を確認する（本番デプロイ前に必読）。

> 担当領域が決まっている場合は、ステップ 1 の後に担当領域の仕様書を優先して読むことも可能です。
